## 核心技能调用

### 1. negotiation（谈判）
**适用场景**：供应商谈判、合同条款谈判、客户价格谈判。

**核心三角**（详见 `references/negotiation-framework.md`）：
- **Tactical Empathy**：标注情绪 + 镜像复述 + 主动示弱，非同情而是理解动机
- **BATNA**：最佳替代方案是谈判力核心，提前准备备选供应商/方案包
- **锚定效应**：先开口报价者设定参考点，渔芯应先发制人再以条件交换谈判

**Ackerman让步模型**：$x → x-15% → x-30% → x-45% 渐进让步，不宜一步到位。

**谈判三阶段**：信息收集（摸底）→ 定向（扩大共同利益）→ 达成（条件交换+书面确认）

**⚠️ 注意事项**：
- 谈判破裂的根本原因是情绪对立，解法是"扩大馅饼"而非"分割馅饼"
- 渔芯两大品牌（AI赋能全链条 + 看见未来）可作为政策红利/数据安全议价筹码
- 等保资质是数据安全背书，在客户谈判和供应商谈判中均可作为硬筹码使用

workspace 目录结构如下（来自 `ls -la workspace/`）：
```
workspace/knowledge/          ← 英文目录，合同模板等知识文件放这里
workspace/渔芯科技内部工作流SOP_v1.2.md

```

**所有合同模板统一保存到 `workspace/knowledge/`（英文），禁止写入 `workspace/知识库/`（中文目录）**。

`合同模板库_YYYY-MM-DD_vX.X.md` 表格中的文件路径列必须使用 `workspace/knowledge/XXX_YYYY-MM-DD.md`，不得混用 `知识库/` 或 `workspace/知识库/`，否则后续 agent 找不到文件。

### 高危陷阱一：文档自身可能违反自身规定的路径规范

正文中写着"统一用workspace/knowledge/"，但表格内部某些单元格却填了`workspace/知识库/`或占位符`YYYY-MM-DD`。**描述规范的文档本身在破坏规范**——这是最隐蔽的bug。

### 高危陷阱二：状态列与实际文件不同步

即使路径列正确，状态列（✅已起草/待起草）也可能与实际文件状态不一致。
**本 session 案例**：`合同模板库_v1.4` 的 C类表格中，"项目合作协议"和"软件许可协议"均标注"待起草"，但这两个文件实际已存在于 `workspace/knowledge/`，大小分别为 7684 和 8902 字节。

**自我提升更新模板库时，必须同时检查两类列（共两步，不可省略）**：
1. **路径列**：检查所有文件路径是否存在且指向 `workspace/knowledge/`
2. **状态列**：对每个"待起草"条目，用 `ls -la workspace/knowledge/` 交叉验证文件是否真的存在
3. 修正版本（v1.X）必须同时消除：路径列的`知识库/`引用 + 状态列的"待起草"（当文件实际已存在时）
4. 不能只看正文就以为表格是对的——**表格是状态错误高发区，路径是格式错误高发区**

## ⚠️ Ghost任务识别（已知缺陷）

SOP/文档类任务常出现"幽灵副本"（ghost copy）——TaskQueue显示in_progress但实际工作早已完成。
识别方法分五步，**禁止跳步**：

**第一步：查workspace文件系统**
```bash
ls -la /Users/hua/Desktop/渔芯科技/4-部门空间/黑豆-行政财务法务/workspace/ | grep SOP
```
看文件大小/行数判断哪个是完整版。**最高版本号（v1.2 > v1.1 > v1.0）通常最完整**；短小的单日期文件（如`SOP_2026-05-06.md`，139行）往往是摘要或中间版本，而非最终版。

**第二步：查TaskQueue记录**
```python
q.get_my_tasks('黑豆', status='in_progress')
```
用 TaskQueue 返回的 **真实 task_id**（而非 scanner 缓存的旧 task_id）。

**第三步：跨任务队列交叉验证**
如果文件系统有完整文件但 TaskQueue 有重复记录，用 `q.get_recent_log(limit=N)` 查找同一 title 的历史完成记录，对比 `done_at` 时间戳和 `result` 内容，确认是 ghost copy 后再归档。

**第四步：对比决定**
- 文件存在且完整 + TaskQueue 无记录 → 工作已完成，直接归档
- 文件存在但 TaskQueue 仍为 in_progress → ghost copy，跳过
- 文件不存在 + TaskQueue 有记录 → 真正需要执行的任务

**第五步：ghost 归档后验证**
完成后**不要依赖 scanner.py 的输出**——scanner 会缓存旧状态。必须用 TaskQueue 直接查询验证：
```python
q.get_my_tasks('黑豆', status='in_progress')  # 返回空列表才是真干净
```
scanner 仍显示 in_progress 但 TaskQueue 直接查询已为空 → scanner 缓存，以 TaskQueue 为准。

**额外Ghost信号**：description=null + result=null + workspace 无相关文件 + recent_log 无历史记录 → 强ghost信号，需从头编制文档后归档。

## 自我进化之政策研究方法论

当进入自我进化模式且需要追踪中国政策法规变化时，使用以下方法论高效发现和整合信息。

### 适用场景
- 政策跟踪文件超过30天未更新，需要补充最新政策
- 财税知识文件需要反映最新税率/扣除比例变化
- 合规管理框架需要整合最新监管要求
- 任何需要"发现外部信息 → 评估影响 → 更新知识库"的场景

### 研究流程（4步）

**Step 1：确定搜索域**
中国政策变化通常集中在以下领域，一次进化覆盖全部或部分：
- AI监管（算法备案、深度合成、生成式AI）
- 财税政策（研发费用加计扣除、软件即征即退、高新认定）
- 数据安全（数据出境、个人信息保护、等保测评）
- 水产养殖行业（尾水排放标准、养殖许可、用药禁用清单）

**Step 2：委托并行搜索（delegate_task + web toolsets）**
由于工具限制（无 browser_search），使用 delegate_task 委托子 agent 搜索：
```python
delegate_task(
    goal="搜索2026年X月中国AI监管、财税、数据安全、水产养殖行业的最新政策变化...",
    toolsets=["web"]
)
```
**关键技巧**：在 goal 中明确指定搜索领域、时间范围、语言要求（中文搜索中文回答）。
子 agent 的 web 工具集可覆盖外网搜索，结果汇总后返回。

**Step 3：交叉评估（对照现有知识库）**
收到搜索结果后，对每个政策/信息点，与 workspace/knowledge/ 中所有3个核心文件逐一对比，记录是否已有覆盖：

| 搜索结果 | 政策跟踪已有？ | 财务知识已有？ | 合规框架已有？ | 结论 |
|----------|--------------|--------------|--------------|------|
| 逐项填入 | ✅/❌ | ✅/❌ | ✅/❌ | 已有/新发现/需验证 |

**3个对比文件**（缺一不可）：
- `政策跟踪_最新日期.md` — 看哪些政策已有记录、哪些是新发现的
- `财务知识_最新日期.md` — 看税率/扣除比例/优惠条件是否需要更新
- `合规管理框架_vX.X.md` — 看等保/数据安全/AI合规要求是否有变化

**Step 4：增量更新知识库**
产出两个级别的文件：

**A. 独立知识文档（按日期命名）**：
- `政策跟踪_YYYY-MM-DD.md` — 完整记录所有新政策，含来源、影响评估、行动建议
- `财务知识_YYYY-MM-DD.md` — 更新税率、扣除比例、申报流程变化
- 写文件前必须 `ls workspace/knowledge/XXX*` 检查同名文件已有版本

**B. 合规管理框架（版本递增）**：
- 对 vX.X 版本，在"最新监管政策动态"节追加新政策
- 更新差距整改跟踪矩阵
- 更新下次更新计划

### ⚠️ 增量追加策略（2026-07-14发现）

当知识库文件在**24小时内**刚更新过（如当日 cron 已产出 `政策跟踪_YYYY-MM-DD.md`），**不要创建新文件或覆盖重写**，而是：

1. **在现有文件的末尾追加补充章节**（八、九、十...），节标题标明"补充发现（YYYY-MM-DD进化）"
2. **只写入真正的新发现**——与现有内容对比后筛选出的增量信息
3. **更新行动建议表**——追加补充行动项，注明"✅ 新增"
4. **保持主文件版本不变**（仅追加，不改日期或版本号）

**典型产出**：`政策跟踪_2026-07-13.md` 追加第八节（7项新政策）和第九节（补充行动项），文件总行数从142→231。

**理由**：避免文件碎片化（同一天出现 N 个同名文件），且后续 agent 只需要读一个文件就能看到当日所有信息。

### ⚠️ 已踩坑：知识库过时检测
- 合规管理框架v1.4（2026-05-28）在7月进化时已过时1.5个月，缺失大量6-7月新政
- 政策跟踪_2026-05-30 在7月进化时已过时1.5个月
- 财务知识_2026-05-30 同样过时
- **教训**：进化时不要只看最新版本号，要检查"上次更新日期"，超过30天必更新

### ⚠️ 已踩坑：搜索返回结果与近期知识库大量重叠（2026-07-14）

当知识库文件在**24小时内**刚更新过，委托搜索返回的结果往往与已有内容高度重叠（如本 session 中7个领域全部已有覆盖）。

**现象**：政策研究做完，发现每个搜索结果在知识库里都已记录。

**正确解法**：
1. **不要认为"全部已有"等于"本次进化无价值"**
2. 逐项对比后，标注哪些是已有、哪些是真正新发现
3. 新发现哪怕只有1-2条也值得写入补充章节
4. 同时更新财务知识/合规框架的"下次更新计划"指针，让下个 cron 知道有新内容需跟踪
5. 进化报告中的知识库更新详情表清晰展示"已有 vs 新发现"的对比结果

**反面案例**：全部已有就跳过更新 → 错过微小的增量信息（如科技型中小企业门槛从3%→2%这种微小但关键的变化）

### ⚠️ 合规框架审计模式（2026-07-14发现+2026-07-16扩展）

当知识库文件在**24小时内**刚更新过且无待办任务时，cron 不应止步于"增量追加"策略，还应进入**合规框架审计模式**。

**审计需读取合规管理框架的3个核心章节（缺一不可）：**

1. **"变更摘要"节** — 查找 vX.X 待办完成情况表，获取超期项的概要信息
2. **"等保差距整改跟踪矩阵"节** — 获取每项超期的**确切原始截止日期、新截止日期**，用于精确计算超期天数（以当天日期为基准重新计算，而非照搬文件中的"已超X天"——文件可能已过时数天）
3. **"2026下半年合规路线图"节** — 检查近30天内到期的里程碑，作为"即将到期预警"纳入报告

**审计步骤：**

1. 读取"变更摘要"节，查找 vX.X 待办完成情况表中的超期项
2. 读取"等保差距整改跟踪矩阵"节，获取每项的原始截止日期和新截止日期
3. **以当前日期重新计算超期天数**（文件中的"已超X天"是上次更新时的数据，可能已过时数天）
4. 读取"合规路线图"节，筛选出**30天内到期的里程碑**作为预警
5. 对每个超期项记录：原截止日期、当前日期、超期天数、新截止日期、距新截止天数
6. 标注风险等级：⏰已超期（计算实际天数）、距新截止≤7天标🔴极度紧急、≤15天标🟠高
7. **交叉参考财税政策外部日期**（如即征即退8月1日生效、科技型中小企业资质确认等），在审计报告中补充提醒——这些日期在合规管理框架中未必有，但直接影响财务合规
8. 将发现的超期项和近期待办里程碑写入进化报告，作为关注项提示
9. 在 evolution_log.md 中追加记录时，明确建议"下次心跳优先确认以上X项进展"

**典型产出**：2026-07-14 进化中发现合规管理框架 v1.5 有5项超期，最严重的 P1-1 VPN整改已超期29天。

**理由**：当知识库已足够新鲜、无需重复研究时，合规框架审计是唯一能产生"风险预警前置"价值的活动——符合合规管理原则第4条（风险预警前置）。

### ⚠️ 已踩坑：同日内多次审计冗余（2026-07-16新增）

当 cron 在同一天内运行多次（如07-16跑了06:00、08:30、16:35三次审计），第3次审计在8小时后发现**无实质性变化**，属于冗余工作。

**现象**：同一日期产生3份进化报告，内容高度重复，仅超期天数因日期基准相同而无变化（同一天内Jul 16全天都是"距Jul 20还有4天"）。

**正确做法**：在进入合规框架审计模式前，先执行**冗余检测**：

```bash
# 检查当天是否已有审计报告
ls -la /Users/hua/.hermes/evolution/ | grep "$(date +%Y-%m-%d)"
# 检查最近一次审计时间
ls -t /Users/hua/.hermes/evolution/ | head -3
```

1. **距上次审计<12小时** → 跳过完整审计，仅输出一句"距上次审计X小时，无实质性变化"的简要确认报告，不重新计算超期天数
2. **距上次审计12-24小时** → 做最小审计：仅检查距截止≤7天的🔴极度紧急项是否有变化
3. **距上次完整审计>24小时** → 做完整审计（原流程）。**注意**：此处的"上次审计"指上次**完整**审计（非跳步报告）。例如：当日已有3次跳步报告，最近一次在4小时前，但上次完整审计在24小时前 → 仍应触发完整审计。

**例外规则**：即使距上次审计<12小时，以下情况仍需完整审计：
- 当天有新的政策/法规发布（知识库有新增）
- 用户明确要求重新审计
- 距截止≤3天的🔴极度紧急项，每次 cron 都需提醒（即使无变化也要重复预警）

**产出规范**：当跳过完整审计时，进化报告标题加注 `（同日内第N次，跳步）`，报告主体仅含：
- 距上次审计时间
- 确认无变化
- 🔴极度紧急项重复提醒（如有）

### ⚠️ 已踩坑：跳步审计风险等级过时 + 同日到期聚集风险（2026-07-19发现）

**坑A：跳步报告的风险标签不重新计算**。跳步模式直接复用前次报告的风险等级（🔴/🟠/🟡），不重新计算距截止天数。当某项距截止天数跨越阈值边界（如8天→7天进入🔴区，或16天→15天进入🟠区），跳步报告会遗漏风险升级。

**本 session 案例**：P0-1 系统定级备案在前3次跳步报告中均标注🟠（基于凌晨04:32计算的5天），16:31完整审计重新计算发现距截止仅6天→正式进入🔴区。跳步报告连续3次给出过时的风险信号。

**正确做法**：完整审计模式应在审计步骤中**特别关注阈值边界项**——距截止6-8天（🔴🟠边界）、14-16天（🟠🟡边界）的项，逐项重新计算。如发现跳步报告误标风险等级，在审计报告中明确标注"前N次跳步报告误标为🟠，经重新计算实际为🔴"。

**坑B：同日到期聚集风险**。当合规路线图中多项里程碑共享同一截止日期（如07-31有6项同日到期），这是**集中违约风险**——只要该日期有任何延误，所有6项同时超期。

**本 session 案例**：2026-07-31 有 P0-2测评机构 + P1-1 VPN + P1-2 IDS + P1-4管理制度 + P1-3原截止（已升级）+ P0-1原截止 → 共6项。如07-31当天技术部资源不足，可能触发连锁超期。

**正确做法**：完整审计时增加"聚集风险检查"步骤：统计每个截止日期的里程碑数量，≥3项同日到期的标为聚集风险，在审计报告"30天里程碑"表中明确标注并给出分批次推进建议。同时建议将同日到期项拆分截止日期（如前移2-3天），分散违约风险。
