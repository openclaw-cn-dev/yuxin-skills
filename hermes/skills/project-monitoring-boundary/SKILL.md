# 项目监控边界规范（华哥 2026-06-15 明确要求）

## 核心原则

**每个项目独立监控自己。跨项目数据流单独建入站/出站监控。**

绝对禁止把"项目A → 项目B 的数据流"做成一个跨项目监控任务 —— 它既不是项目A的内部健康，也不是项目B的内部健康，问题定位时责任不清。

## 设计规则

### 每个 cron 任务的 prompt 必须明确写

1. **角色定位表** —— 三栏：监控对象/谁负责/不归你管
2. **检查清单** —— 只列自己项目范围内的检查项
3. **汇报规则** —— 明确"只汇报 X 的状态，绝不提及 Y"
4. **关键提醒** —— 明确"不要调 Y 的 API / 不要重启 Y 的容器"

### 数据流归属原则

- 推送方（数据出口）出问题 → 推方项目监控报警
- 接收方（数据入口）出问题 → 收方项目监控报警
- 数据流本身（速率/积压/失败）→ 在**收方入口**建监控（数据"已经推过来但没正常入库"是收方问题）

```
  POST /api/v1/external/documents

```

- FindEra 自己的健康 → `35a45d2f7355`
- RKR 自己的健康 → `1262b15958f4`
- FindEra 推到 RKR 的数据流 → `8290659f8b28`（RKR 入口端）

## 验证清单（每次新加跨项目监控前）

- [ ] 这个任务名是否清晰表明它属于哪个项目？
- [ ] prompt 里是否明确"不归你管"的边界？
- [ ] 检查项是否只触碰自己项目的资产？
- [ ] 异常告警里是否只包含本项目指标？
